迷宫一样的this指向问题

标签（空格分隔）： js

　　this是个让人又爱又恨的存在。
　　爱它的方便简洁，恨它的复杂多变。
　　我发现，在任何领域，都是学的越少，越觉得不过如此；学的越多，越觉得自己无知。
　　最早接触this，是在函数内部使用，觉得就是替代一下函数而已，不是什么复杂的东西。结果，越到后来，越觉得自己浅薄而无知。对事物下定义过于轻率。
　　
　　今天写下这个总结，将已经出现的this情况大致归总一下，与大家共勉。

　　1.首先我们来说说，this是什么？
　　　　this是个关键字，语言规范里规定他指向函数执行时的当前对象。它代表函数运行时，自动生成的一个内部对象，只能在函数内部使用。

　　2.然后，最主要的是，this指向谁？
　　　　this的指向是它最复杂的地方。需要好好说道说道。
　　大体上来分，this的出现分为两个地方：
　　　　（1）全局作用域，在全局作用域中，this指向window。
　　　　（2）函数中，在函数中this指向当前有名函数，没有有名函数的时候，this指向全局对象window。

　　同样是在函数中，因调用模式不同仍然有不同情况。在JS中调用函数的模式可以分为五种，方法调用模式、函数调用模式、构造函数模式、定时器和匿名函数apply调用模式。这些模式在如何初始化关键参数this上存在差异。
　　
　　1.方法调用模式:　　谁调用指向谁
　　　　函数作为某个对象的成员方法调用时this指向该对象。
　　2.函数调用模式:　　指向window
　　　　当一个函数并非一个对象的属性时，那么它就是被当做函数来调用的。在此种模式下，this被绑定为全局对象，默认指向了全局对象window。
　　3.构造函数模式：　　指向构造函数新建的对象
　　　　函数作为构造函数调用时this指向用该构造函数构造出来的新对象。
　　4.以及匿名函数：指向window
　　　　setTimeout和setInterval定时器运行及匿名函数调用随时，this指向全局对象window。

　　再然后，重头戏来了！
　　如果只有上面那些，都不算大问题，大不了多背几遍，那都不叫事儿。
　　问题是还有下面这些！

　　apply、call、bind。
　　
　　这三个函数是函数对象的一个方法，他们的作用就是为了改变函数执行时候的this指向，具体用法如下：

　　call：call(obj[,arg1][,arg2]);第一个参数为强制改变需要指向的对象，后边可选的是该函数的参数，如果不传obj的话默认为window。

　　apply:apply(obj[,arr]);第一个参数为强制改变需要指向的对象，后边可选的是该参数集合的数组形式，如果不传obj的话默认为window。

　　apply和call的作用和调用形式基本一致，不同的是call后面的参数与方法中是一一对应的，而apply的第二个参数是一个数组，数组中的元素是和方法中一一对应的，这就是两者最大的区别。两者都可以不传参数，此时默认改变指向的对象为全局对象。

　　bind:bind的调用形式和call相同，但是他返回的是改变调用对象后的函数引用，所以还要再执行一次，也就是obj.fun().bind()()。

总结:this作为函数运行时，自动生成的一个内部对象，只能在函数内部使用。具体this指向谁，要看this的所有函数是谁调用的，具体情况可分为全局作用域、作为某对象的方法调用、直接执行、匿名函数直接执行、call、apply、bind强制改变调用对象等。

　　明天上实例，给大家绕一下，看看有没有入坑的。拜拜喽~~~

JS中的面向过程和面向对象

标签（空格分隔）： js面向对象

  直到接触面向对象，才真正理解，编程分为面向过程和面向对象两种，同样是编程，却有着很大的不同。

  借用网友的举例，来简单概括一下。

  面向过程就是分析出解决问题所需要的步骤，然后用函数把这些步骤一步一步实现，使用的时候一个一个依次调用。

  面向对象是把构成问题的事物分解成各个对象。建立对象的目的不是为了完成一个步骤，而是为了描述某个事物在整个解决问题的步骤中的行为。

  以五子棋游戏举例说明。
  面向过程的设计思路就是首先分析问题的步骤：
    1、开始游戏，
    2、黑子先走，
    3、绘制画面，
    4、判断输赢，
    5、轮到白子，
    6、绘制画面，
    7、判断输赢，
    8、返回步骤2，
    9、输出最后结果。
  把上面每个步骤用分别的函数来实现，问题就解决了。

  而面向对象的设计则是从另外的思路来解决问题。
  整个五子棋可以分为：
    1、黑白双方，这两方的行为是一模一样的，
    2、棋盘系统，负责绘制画面，
    3、规则系统，负责判定诸如犯规、输赢等。
  第一类对象（玩家对象）负责接受用户输入，并告知第二类对象（棋盘对象）棋子布局的变化，棋盘对象接收到了棋子的变化就要负责在屏幕上面显示出这种变化，同时利用第三类对象（规则系统）来对棋局进行判定。

  简单来说，面向过程是一件事“该怎么做”，面向对象是一件事“该让谁来做”，然后那个“谁”就是对象，他要怎么做就是我们怎么写来决定的。最后一群对象合力能把事做好就行了。

飞流直下瀑布流

标签（空格分隔）： js

  瀑布在地质学上叫跌水，即河水在流经断层、凹陷等地区时垂直地从高空跌落的现象。在河流存在的时段内，瀑布是一种暂时性的特征，它最终会消失。

  瀑布流，又称瀑布流式布局。是比较流行的一种网站页面布局，视觉表现为参差不齐的多栏布局，随着页面滚动条向下滚动，这种布局还会不断加载数据块并附加至当前尾部。最早采用此布局的网站是Pinterest，逐渐在国内流行开来。国内大多数清新站基本为这类风格。

  虽是一字之差，却是完全不同的两个东西，一个是大自然鬼斧神工，一个是程序员们的鬼斧神工。

  瀑布流布局的效果以满屏图片为主，一些小清新的网站多数以此布局。操作简单，在浏览网站的时候，用户轻轻滚动鼠标滚轮就可以浏览美妙的图片。以花瓣网为例，花瓣网就是使用了瀑布流布局，整体效果清新明了，便于浏览。
http://huaban.com/

  子网页中，另外加了根据浏览区域宽度动态生成列数的模式，每次宽度改变，图片排列顺序会因为列数改变而改变。改变的基础是寻找当前高度最低的图片，将新的图片生成在它下面，依次执行。
http://huaban.com/boards/13707285/
  这样的操作带来的视觉效果就是在宽度改变的时候，图片都乖乖的，各找其位，从不错乱，自动排序的效果也带来错落有致的优雅。

  这实在是个相当好的布局模式。不过当然，再好的东西也不会是完美无缺，总是有个优劣利弊，这就来说说它的优缺点。

布局优点：
  1.有效的降低了界面复杂度，节省了空间：我们再也不需要臃肿复杂的页码导航链接或按钮了。
  2.对触屏设备来说，交互方式更符合直觉：在移动应用的交互环境当中，通过向上滑动进行滚屏的操作已经成为最基本的用户习惯，而且所需要的操作精准程度远远低于点击链接或按钮。
  3.更高的参与度：以上两点所带来的交互便捷性可以使用户将注意力更多的集中在内容而不是操作上，从而让他们更乐于沉浸在探索与浏览当中。

  当然，除了优点，剩下的就是缺点：

  1. 有限的用例：
      无限滚动的方式只适用于某些特定类型产品当中一部分特定类型的内容。
      例如，在电商网站当中，用户时常需要在商品列表与详情页面之间切换，这种情况下，传统的、带有页码导航的方式可以帮助用户更稳妥和准确的回到某个特定的列表页面当中。
  2. 额外的复杂度：
      那些用来打造无限滚动的JS库虽然都自称很容易使用，但你总会需要在自己的产品中进行不同程度的定制化处理，以满足你们自己的需求;另外这些JS库在浏览器和设备兼容性等方面的表现也参差不齐，你必须做好充分的测试与调整工作。
  3. 再见了，页脚：
      如果使用了比较典型的无限滚动加载模式，这就意味着你可以和页脚说拜拜了。
      最好考虑一下页脚对于你的网站，特别是用户的重要性;如果其中确实有比较重要的内容或链接，那么最好换一种更传统和稳妥的方式。
      千万不要耍弄你的用户，当他们一次次的浏览到页面底部，看到页脚，却因为自动加载的内容突然出现而无论如何都无法点击页脚中的链接时，他们会变的越发愤怒。
  4. SEO：
      集中在一页当中动态加载数据，与一页一页的输出相比，究竟那种方式更利于SEO，这是你必须考虑的问题。对于某些类型网站来说，在这方面进行冒险是很不划算的。
  5. 关于页面数量的印象：
      其实站在用户的角度来看，这一点并非负面;不过，如果对于你的网站来说，通过更多的内容页面展示更多的相关信息(包括广告)是很重要的策略，那么单页无限滚动的方式对你并不适用。

  所以说，任何东西都是双刃剑，用得好，缺点也是优点，用不好，优点也是缺点。

  今天瀑布流就聊到这了，多谢捧场，拜拜~~~

我和AJAX的纠结之旅

标签（空格分隔）： 未分类

  连续过了三天一头雾水的日子。
  AJAX这个小东西，让人云里雾里了好几天。今天就来说说它吧。顺便也记录下来给未来的我一些提示，免得再次遇到感觉似曾相识，却还是云里雾里不清不楚。

  Ajax 即“Asynchronous Javascript And XML”。异步 JavaScript 和 XML，是指一种创建交互式网页应用的网页开发技术。通俗来说，它是一种在无需重新加载整个网页的情况下，能够更新部分网页的技术，现在应用极为广泛。不刷新就能拿到新的数据？原来的网页那是想想都觉得不可能的，所有的数据更新都只能是在刷新之后，或者打开新页面跳转之后。

  这样的技术发展无疑给了用户更好的浏览体验。不过我个人觉得，这也使用户在不知不觉中将更多的时间花在了互联网上，日复一日，年复一年的在庞大新奇的新信息中翱翔，不知今夕是何夕。只要有能联网的设备，便有一种任你风云纵横，天下信息尽在我手，养成以上帝视角俯瞰他人的习惯。

  扯得有点远，继续回来说AJAX。

  ajax是由一系列老技术组成的不算新技术的技术，最早是由google将之发扬光大的，google将它用在搜索框上，实现了我们现在已经习以为常的联想关键词搜索。

  当然，任何一项技术都有利弊之说，ajax也不例外，大致列举了几项，如有问题，欢迎指出。

  ajax优点:
    1.减轻服务器的压力，按需获取数据,最大程度的减少冗余请求；
    2.局部刷新页面,减少用户心理和实际的等待时间,带来更好的用户体验；
    3.基于xml标准化,并被广泛支持,不需安装插件等；
    4.进一步促进了页面和数据的分离。

  ajax缺点:
    1.AJAX大量的使用了javascript和ajax引擎。
      这些取决于浏览器的支持，在编写的时候需要考虑对浏览器的兼容性；
    2.AJAX只是局部刷新,所以页面的后退按钮是没有用的。

  最后列举一个ajax的交互模型给大家，便于参考：

  ajax交互模型（打电话模式）
  1.有个电话
    new XMLHttpRequest
  2.给谁打（输入号码）
    ajax.open(‘get’,url,true);
  3.发送号码
    ajax.send()
  4.需要等待
    ajax.onload
  5.通话
    ajax.responseText

  好了，今天就写到这里了，已经过了午夜12点了。12点之前我没有变成公主，自然也没有水晶鞋和南瓜马车带我去参加华丽的舞会。所以我依然是个灰姑娘，只能乖乖的一直敲代码直到现在。
  自从开始认认真真的努力学习，就开始觉得一天二十四个小时根不能不够用嘛，明天见喽各位~~~

在此处输入标题

标签（空格分隔）： 未分类

火星文–正则

火星文你好，久仰大名，如今幸会，深感荣幸。
早已耳闻火星文，如今才算是认认真真的实际接触，感觉果然名不虚传。看的时候各种懂，写的时候也还行，然后两天之后拿来看，一脸懵，这是啥玩意儿？
哈哈，闲话不多说，我们来掰扯掰扯这火星文。

正则，又叫正则表达式，是对字符串操作的一种逻辑公式。就是用事先定义好的一些特定字符、及这些特定字符的组合，组成一个“规则字符串”，这个“规则字符串”用来表达对字符串的一种过滤逻辑。
目前主要的应用场景在登录框方面。用以验证用户输入的信息是否符合规则。

正则的书写格式：
简写格式为： /字符串/ ，两个斜杠中间只能为字符串，并且不能加引号；
标准写法为：new Regexp (正则 || 字符串) 。

两种方法的区别为：简写中的字母会被认为是字符串，放不了变量。

这里列举一下我们经常用的一些元字符，如果有人需要完整的元字符表，可以点击链接https://baike.baidu.com/item/%E6%AD%A3%E5%88%99%E8%A1%A8%E8%BE%BE%E5%BC%8F/1700215?fr=aladdin
也可以直接自己百度哦。

\d      一个数字
\D      一个非数字
\w      一个字母数字下划线
\W      一个非字母数字下划线
\s      一个空格
\S      一个非空格
^       开始
&       结尾
（）    子项
【】    在中括号中取其一，中括号自带或属性
{a,b}   a代表最小，b为最大，如{0,5}  最小没有最多5个
+  {1,}     最小一个最多不限
?  {0,1}    最小可以没有，最多只能有一个
*  {0,}     最小没有最多不限

使用这些简单的元字符我们能做到什么呢？
来来来，见证奇迹的时候到了！！

1.QQ号验证：

let str = '102111234567';
alert(/^[1-9]\d{4,11}$/.test(str)?'是QQ':'不是QQ');

2.邮箱格式验证：

   let str = 'x1236232@qq.com.cn';
    alert(/^[1-9]\d{4,11}|^[A-Za-z]\w{5,17}@[0-9a-z]{2,8}(\.com|\.cn|\.net){1,2}$/.test(str)?'是':'不是');

3.身份证验证：

let str = '430112196905171335';
console.log(str.match(/^[1-9]\d{5}(1[89][0-9][0-9]|2[0-3][0-9][0-9])(0[1-9]|1[0-2])((0[1-9])|([12][0-9])|(3[01]))\d{3}[\dxX]$/));

这几个是比较常用的一些小应用，与大家共勉。
